üõë Before Escalating - Have You Completed These Steps?

0

Master the Basics First!

Understand how our products work correctly. This foundational knowledge is your superpower and helps you quickly spot when something isn't right.

1

Hear the Customer's Story

Actively listen to your customer. What's their core problem? Get the initial issue clear in your mind.

2

Ask Smart Questions

Put on your detective hat and ask clarifying questions. Is it "call disconnected" or "loss of audio on hard phone in single location, consistently"? The more precise you are, the faster you'll find the solution.

3

Check the Settings, Like a Pro

Check standard product settings AND customer-specific settings. Look for anything contributing to the problem and ensure everything is configured correctly.

4

Document Everything (Your Future Self Will Thank You!)

ALWAYS document every example of the problem and all associated logs. Even if the issue resolves itself, this data is invaluable if it resurfaces.

5

Fix It or Escalate It

Determine if the issue can be resolved with a configuration change or if something is genuinely broken. Then either fix it yourself or escalate with all your detailed findings.

What It Is:

A concise, technical statement of the actual underlying problem, stripped of customer-specific language. Think of it as the core technical malfunction you've identified.

Why It's Important:

This helps Tier 3 immediately grasp the technical issue without sifting through conversational details. This should be your "one-liner" diagnosis.

‚ùå Bad Examples:

‚Ä¢ "Customer is complaining about disconnected calls"
‚Ä¢ "Loss of audio"
‚Ä¢ "Phone not working"
‚Ä¢ "Customer upset about calls"
‚Ä¢ "Issues with phone system"

‚úÖ Good Examples:

‚Ä¢ "Sporadic call disconnections occurring during outbound PSTN calls"
‚Ä¢ "One-way audio observed on hard phones after 5 minutes of call time"
‚Ä¢ "SIP registration failures on Polycom VVX 500 phones using TLS"
‚Ä¢ "Inbound calls to DID +1-555-123-4567 receive fast busy (SIP 503)"

What It Is:

Essential context that frames the problem. This includes what the customer is experiencing, when it started, and any relevant environmental factors.

Why It's Important:

It connects the technical problem back to the customer's real-world impact and provides crucial environmental context.

‚úÖ Good Background Example:

Customer is reporting frequent call disconnections. This issue began approximately 3 days ago after a new network firewall was implemented at their main office. Affects users on both hard phones (Polycom VVX 500) and soft clients (RC Desktop App v21.3.20). Impact: 15 users in Denver office, approximately 30-40% of outbound calls affected. Inbound calls work normally. No changes to RC account settings during this timeframe. Customer initially thought it was carrier issue but testing shows calls disconnect at exactly 5-minute mark consistently.
üí° Things to Include in Background:
  • Timeline: When started, duration, frequency
  • Scope: Number of users, locations, devices affected
  • Recent changes: Network, firmware, configuration, ISP
  • Patterns: Time of day, specific destinations, call types
  • Customer environment: Office/remote, network type, VLAN setup

What It Is:

Precise, repeatable instructions that allow anyone to consistently trigger the problem. Include all necessary details like login credentials, specific settings, and sequences of actions.

Why It's Important:

Reproducing the issue is often the fastest way to diagnose it. This saves countless hours for Tier 3.

‚úÖ Good Steps to Reproduce - Call Issue:

A. Using Polycom VVX 500 (firmware 5.9.7.3480) registered to extension 101 B. From Denver office network (IP: 203.0.113.45) C. Dial outbound to +1-720-555-1234 (external PSTN number) D. Wait for call to connect and answer E. Establish conversation for 5 minutes F. At approximately 4:58-5:02 minute mark, call disconnects G. Phone displays "Call Ended" - no error code H. Reproduces on 8 out of 10 test calls

‚úÖ Good Steps to Reproduce - Web/Config Issue:

A. Login to RingCentral Admin Portal as admin@company.com B. Navigate to Phone System ‚Üí Auto-Receptionist ‚Üí Main Number ‚Üí IVR Menu C. Click "Upload Custom Prompt" D. Select file: greeting_update.wav (attached - 2.1MB, 16-bit, 8kHz mono WAV) E. Click "Upload" button F. Error appears: "Upload failed: Invalid file format (Error Code: 403-B)" G. File does not appear in prompt list H. Tested with both WAV and MP3 formats - same error
üö® Steps to Reproduce Must Include:
  • Exact sequence of actions (A, B, C format)
  • Specific devices, extensions, numbers involved
  • Network location (if relevant)
  • Time/duration factors (e.g., "after 5 minutes")
  • Consistency (every time vs intermittent)
  • Any required file attachments referenced

What It Is:

Clearly state what SHOULD happen if the system were working correctly when you follow the steps.

Why It's Important:

This provides a baseline for comparison and confirms your understanding of the product's intended functionality.

‚úÖ Good Expected Behavior - Call:

Call should remain connected for the duration of the conversation until either party hangs up. No automatic disconnection should occur at any time interval. NAT timeout should be maintained by SIP keepalive packets.

‚úÖ Good Expected Behavior - Upload:

The audio prompt greeting_update.wav should successfully upload, appear in the IVR prompt list with filename and duration displayed, and be immediately selectable for use in the IVR menu.

‚úÖ Good Expected Behavior - Registration:

Phone should send SIP REGISTER to sip.ringcentral.com, receive 401 Unauthorized with nonce, respond with MD5 hash, and receive 200 OK registration confirmation. Phone status should display "Registered" or extension number on screen. Phone should maintain registration with periodic re-registration every 3600 seconds.

What It Is:

An exact description of what ACTUALLY happened when you followed the "Steps to Reproduce." Include any error messages, specific symptoms, or unexpected outcomes.

Why It's Important:

This confirms the problem is observable and provides key data points for diagnosis.

‚úÖ Good Actual Behavior - Call Issue:

Call connects successfully and audio is bidirectional for approximately 4 minutes 58 seconds. At 5:00 mark (¬±2 seconds), call abruptly disconnects. Phone displays "Call Ended" with no error code. No audio degradation prior to disconnect - immediate silence. Tested 10 times: - 8 calls disconnected at 4:58-5:02 mark - 2 calls completed normally (30+ minutes) - Pattern consistent across 3 different phones - Only affects outbound calls, inbound calls normal

‚úÖ Good Actual Behavior - Error Message:

After clicking "Upload" button, system displays error message within 2-3 seconds: "Upload failed: Invalid file format (Error Code: 403-B)" The file does not appear in the prompt list. Browser console shows HTTP 403 Forbidden response from provisioning server. Attempted with: - WAV file (16-bit, 8kHz, mono) - same error - MP3 file (128kbps) - same error - Different file sizes (500KB to 3MB) - same error Same behavior in Chrome, Firefox, and Edge browsers.

üí° How to Capture HAR File:

  • 5. Right-click anywhere in Network tab ‚Üí "Save all as HAR with content"
  • 6. Save the .har file and attach to case
üí° How to Capture Console Logs:

  • 4. Reproduce the issue

  • 6. Include ALL red errors and yellow warnings

üí° Good Troubleshooting Shows:

  • Data gathering (measurements, tests, logs)
